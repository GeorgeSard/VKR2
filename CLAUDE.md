## 1. Тема и постановка задачи

**Тема ВКР:** «Разработка модели искусственного интеллекта для прогнозирования задержек авиарейсов и классификации их причин».

**Тип работы:** магистерская ВКР по программе машинного обучения (программа транспортной направленности).

**Решаемые ML-задачи (две головы):**
- **Задача A — бинарная классификация / регрессия**: будет ли рейс задержан / на сколько минут.
- **Задача B — многоклассовая классификация причины задержки** (погода, технические причины, ATC/УВД, безопасность, авиакомпания, late aircraft и т.д.).

**Тип данных:** табличные → стек строится вокруг классического ML и градиентного бустинга, **не** вокруг глубоких нейросетей.

**Возможные источники датасетов:**
- BTS On-Time Performance Data (US DOT) — основной кандидат, миллионы записей.
- Kaggle Flight Delays / US Flights datasets.
- При необходимости — обогащение метеоданными (NOAA, METAR-архивы) как отдельный feature engineering эксперимент.

**Транспортная задача — обязательное требование:** руководитель явно сказал, что в комиссии будет человек из транспорта, и распознавание «кошки/собаки» вызовет смех. Авиационные задержки — **точно проходит**.

## 2. Философия научного руководителя (это закон проекта)

Прямые формулировки руководителя, которые **формируют все решения** в проекте:

1. **«Вы не учёный, вы инженер, который выстраивает Machine Learning процесс».** Главный результат ВКР — выстроенный процесс, а не модель.
2. **«Если вы не построили цикл — вы ничего не сделали».** Цикл = данные → DVC → feature engineering → обучение → подбор гиперпараметров → внедрение → мониторинг → обратная связь → новые данные.
3. **«Качество модели определяется не вашим скилом подбора, а реальной жизнью».** Поэтому 60–70% точности — это нормально, если процесс позволяет инкрементально улучшаться.
4. **«Garbage in — garbage out»** — поэтому работа с датасетом критична: версионирование, аннотация по стандарту, EDA, контроль баланса классов.
5. **«Повторяемость — ключевой признак того, что вы построили процесс».** Любой эксперимент должен быть воспроизводим: тот же датасет (по версии DVC) + тот же код = те же метрики.
6. **«Архитектуру создавать не надо, если в этом нет смысла».** Берём готовые алгоритмы (XGBoost, LightGBM, CatBoost, RandomForest, LogReg, ансамбли). Никаких самописных нейросетей.
7. **«Эксперименты — это просто кнопки нажимать».** Если процесс автоматизирован через MLflow + DVC, эксперименты — это просто запуск пайплайна с разными параметрами.
8. **«Программного обеспечения должно быть минимум».** Внедрение = FastAPI с ~4 ручками. Не нужно делать веб-приложение, фронтенд, БД с авторизацией и т.д. Фокус — на ML-процессе.
9. **«Mlflow поднимается тремя docker-контейнерами»** — вся инфраструктура должна быть контейнеризована.
10. **«Гипотеза должна быть доказана».** Если хочешь утверждать, что проблема в архитектуре — покажи, что менял датасет, и это не помогало. И наоборот.

## 4. Жизненный цикл ML (это священное)

Каждый раз, когда Claude генерирует код, он явно указывает, **какой этап цикла закрывается**.

```
┌─────────────────────────────────────────────────────────────┐
│ 1. Domain analysis  →  2. Data collection & annotation      │
│        ↑                         ↓                          │
│ 8. Feedback loop    ←   3. Data versioning (DVC)            │
│        ↑                         ↓                          │
│ 7. Monitoring       ←   4. Feature engineering              │
│        ↑                         ↓                          │
│ 6. Deployment       ←   5. Training + hyperparam tuning     │
│   (FastAPI+Docker)         (MLflow tracking)                │
└─────────────────────────────────────────────────────────────┘
```

**Презентация ВКР будет строиться по этим этапам**, поэтому каждый этап должен иметь:
- свой коммит/набор коммитов в git;
- свой артефакт (датасет в DVC, эксперимент в MLflow, образ Docker и т.д.);
- свою демонстрацию в дашборде / графике / таблице сравнения.

## 5. Технологический стек (фиксированный)

### 5.1 Обязательное (требования руководителя)

| Категория | Инструмент | Зачем |
|---|---|---|
| Версионирование кода | **Git + GitHub/GitLab** | стандарт |
| Версионирование данных и моделей | **DVC** | прямое требование руководителя |
| Трекинг экспериментов и реестр моделей | **MLflow** | прямое требование руководителя |
| Сервис инференса | **FastAPI** | прямое требование руководителя |
| Контейнеризация | **Docker + docker-compose** | «3 контейнера» |
| Мониторинг (observability) | **Prometheus + Grafana** (минимум) либо логирование + custom dashboards | прямое требование |

### 5.2 ML-стек (выбран под табличные данные)

- **Python 3.11+**
- **pandas, numpy, polars** — обработка данных
- **scikit-learn** — baseline-модели, метрики, препроцессинг, pipeline
- **XGBoost, LightGBM, CatBoost** — основные кандидаты на победу (бустинги исторически выигрывают на табличке)
- **Optuna** — подбор гиперпараметров (предпочтительно, т.к. умнее GridSearch)
- **imbalanced-learn** — для дисбаланса классов причин задержек
- **shap** — интерпретация фичей (это даёт научную ценность для главы 3)

### 5.3 Качество кода

- **uv** или **poetry** — менеджер зависимостей и виртуального окружения
- **ruff** — линтер + форматтер
- **mypy** — статическая типизация (где разумно)
- **pytest** — тесты
- **pre-commit** — хуки на коммиты

### 5.4 ⛔ Что ЗАПРЕЩЕНО (прямой запрет руководителя)

- **LLM в любом виде**: ни fine-tuning, ни prompt-engineering, ни агенты, ни RAG.
- **Готовые предобученные модели как финальный результат** — мы должны обучить свои модели на своём датасете.
- **Самописные нейросетевые архитектуры** — «вы не математик и не архитектор моделей».
- **Jupyter-only решения** — ноутбуки только для EDA и для глав 1–2. Основной код — модули в `src/`.
- **Фронтенд / красивый UI** — программного обеспечения должно быть **минимум**.

## 8. Конкретные правила работы Claude

### 8.1 На каждом ответе

1. Указать, **на каком этапе плана** мы сейчас находимся (например: «Этап 6: MLflow и базовое обучение»).
2. Указать, **какой шаг ML lifecycle** закрывается этим действием.
3. Если действие не вписывается в план — сначала **обсудить с пользователем**, потом делать.

### 8.2 Git-дисциплина

- Инициализировать git **в самом начале** (если ещё не инициализирован).
- Делать `.gitignore` сразу: `data/`, `models/`, `mlruns/`, `.venv/`, `__pycache__/`, `*.pyc`, `.dvc/cache`, `.env`, `.idea/`, `.vscode/`.
- Коммиты — в формате **Conventional Commits**: `feat:`, `fix:`, `docs:`, `chore:`, `experiment:`, `refactor:`.
- После каждого логического блока работы — **предложить коммит** и сформулировать сообщение.
- Эксперименты — в отдельных ветках `experiment/<short-name>` (например `experiment/optuna-xgboost`).
- Никогда не коммитить большие файлы (датасеты, модели) — только через DVC.

### 8.3 Что Claude делает САМ без спроса

- Создаёт нужные папки и пустые `__init__.py`.
- Добавляет docstrings и type hints в новый код.
- Запускает линтер на новом коде (если есть `bash`).
- Обновляет `requirements.txt` / `pyproject.toml` при добавлении новой зависимости.
- Обновляет `params.yaml`, когда вводится новый гиперпараметр.

### 8.4 Что Claude делает только с подтверждением

- Удаляет файлы.
- Делает rebase / force push / переписывает историю git.
- Меняет уже залогированные эксперименты в MLflow.
- Меняет версии данных в DVC remote.
- Изменяет `CLAUDE.md` (этот файл).

### 8.5 Метрики, которые мы трекаем в MLflow

**Для задачи A (есть/нет задержка):** `accuracy`, `precision`, `recall`, `f1`, `roc_auc`, `pr_auc`, `confusion_matrix`.

**Для задачи A (минуты задержки, регрессия):** `mae`, `rmse`, `r2`, `mape`.

**Для задачи B (причина задержки, multi-class):** `accuracy`, `macro_f1`, `weighted_f1`, `per-class precision/recall`, `confusion_matrix`.

**Системные:** `train_time_seconds`, `model_size_mb`, `inference_latency_p50/p95/p99`.

**На уровне моделирования:** также логируем версию датасета (DVC hash), версию кода (git sha), random seed.

### 8.6 Принципы воспроизводимости

- Все random seeds — в `params.yaml`, переиспользуются везде.
- Каждая модель в MLflow тегается: `git_commit`, `dvc_data_version`, `feature_set_name`.
- В отчёте по эксперименту обязательно есть команда для воспроизведения: `dvc repro -f <stage>` либо `python -m src.models.train --config configs/exp_07_3.yaml`.

## 9. Чеклист готовности к защите

Перед защитой проект должен **демонстрируемо** показывать:

- [ ] `git log --oneline` — видна история всех этапов жизненного цикла.
- [ ] `dvc dag` — виден полный пайплайн данных → модель.
- [ ] MLflow UI — десятки запусков с разными конфигурациями, есть очевидный «лучший».
- [ ] `docker compose up` — поднимается mlflow + api за одну команду.
- [ ] `curl POST /predict/delay` — возвращает прогноз с probability.
- [ ] Grafana / лог — видны накопленные предикты с уверенностями.
- [ ] `reports/` — графики влияния каждого этапа на качество.
- [ ] Сформулированная и доказанная гипотеза о том, где основная точка роста (данные / признаки / модель).

Если хотя бы один пункт не выполнен — по словам руководителя, «вы ничего не сделали».
