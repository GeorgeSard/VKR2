# Инструкция к датасету `scored_test_dataset.*`

Это **финальный датасет с прогнозами обеих моделей на тестовой выборке**
(рейсы с **июля 2025 года** и позже — модели их никогда не видели при
обучении). Сгенерирован скриптом `python -m src.models.score_dataset`.

## Зачем он нужен

В одном файле собраны:
1. **Реальные данные** про каждый рейс из тестовой выборки.
2. **Что предсказала модель A** — была ли задержка > 15 минут.
3. **Что предсказала модель B** — какая причина задержки.
4. **Колонки-маркеры** (`✓` / `✗`), которые сразу показывают, **где
   модель угадала, а где промахнулась**. Можно открыть в Excel и
   отфильтровать только ошибки — это удобно для отчёта.

## Файлы

В папке `reports/`:

| Файл | Размер | Чем удобен |

| `scored_test_dataset.xlsx` | ~4 MB | Excel. **Три листа**: `scored_test` (всё), `preview_first_100` (первые 100 строк), `cause_confusion` (матрица «факт × прогноз» для причин) |

Всего **36 983 строки** — по одной на каждый рейс из тестовой выборки.

## Описание колонок

Колонки идут именно в этом порядке — слева направо.

### 1. Идентификаторы рейса

| Колонка | Что это | Пример |
|---|---|---|
| `flight_id` | Уникальный ID рейса в датасете | `RU202507010183018` |
| `flight_date` | Дата вылета (по плану) | `2025-07-01` |
| `airline_code` | IATA-код авиакомпании | `SU`, `S7`, `5N` |
| `airline_name` | Название авиакомпании | `Аэрофлот`, `S7 Airlines` |
| `flight_number` | Номер рейса | `SU1775` |

### 2. Маршрут и расписание

| Колонка | Что это | Пример |
|---|---|---|
| `origin_iata` | Код аэропорта вылета | `SVO` |
| `origin_city` | Город вылета | `Москва` |
| `destination_iata` | Код аэропорта прилёта | `KRR` |
| `destination_city` | Город прилёта | `Краснодар` |
| `scheduled_departure_local` | Плановое время вылета (местное) | `2025-07-01 05:25:00` |
| `distance_km` | Расстояние, км | `1222` |
| `aircraft_family` | Семейство ВС | `A321`, `B738`, `SSJ100` |

### 3. Что произошло на самом деле (факт)

Эти колонки — **ground truth**, то есть правда. В реальной системе
прогнозирования их **не было бы** на момент предсказания, мы их видим
только потому, что тест-выборка — историческая.

| Колонка | Что это | Возможные значения |
|---|---|---|
| `actual_delay_minutes` | Реальная задержка вылета, минуты | `0.0`, `24.0`, `124.0` |
| `actual_is_delayed` | Реально ли рейс задержан > 15 мин | `вовремя`, `задержан`, `—` (если рейс отменён) |
| `actual_cause` | Реальная причина задержки | `без задержки`, `погода`, `авиакомпания (операционные)`, `каскадная (от прошлого рейса)`, `загруженность аэропорта`, `отменён`, `безопасность` |

### 4. Прогноз модели A — задержка / нет

| Колонка | Что это |
|---|---|
| `predicted_delay` | **Прогноз модели**: `вовремя`, `задержан` или `—` (для отменённых рейсов вылета не было — прогноз не имеет смысла) |
| `delay_probability` | **Уверенность модели в задержке**, число от 0 до 1. `0.85` = «модель почти уверена, что задержка будет», `0.30` = «скорее не задержится». Порог принятия решения = **0.5** |

### 5. Прогноз модели B — причина задержки

| Колонка | Что это |
|---|---|
| `predicted_cause` | **Прогноз модели**, одна из 7 причин (те же значения, что в `actual_cause`) |
| `cause_confidence` | **Уверенность модели в этой причине**, от 0 до 1. Чем ближе к 1 — тем модель «уверена», что причина именно эта (а не любая из остальных шести). Например, `0.66` означает 66 % вероятности этой причины |

### 6. Маркеры правильности — главное для отчёта

| Колонка | Что это | Возможные значения |
|---|---|---|
| `binary_correct` | Угадала ли модель A | `✓` (угадала), `✗` (ошиблась), `н/п` (рейс отменён, прогноз неприменим) |
| `cause_correct` | Угадала ли модель B | `✓` (угадала), `✗` (ошиблась) |

## Как читать датасет в Excel — типовые сценарии

### Сценарий 1: Посмотреть, где модель A ошиблась

1. Открыть `scored_test_dataset.xlsx`, лист `scored_test`.
2. **Данные → Фильтр** на первой строке.
3. В колонке `binary_correct` оставить только `✗`.
4. Получишь все рейсы, где модель сказала «вовремя», а на самом деле
   была задержка (или наоборот).

### Сценарий 2: Найти самые «уверенные» ошибки модели B

1. Отфильтровать `cause_correct = ✗`.
2. Отсортировать по `cause_confidence` по убыванию.
3. В верху списка — рейсы, где модель **уверенно ошиблась**. Это
   самые интересные кейсы для разбора в отчёте: модель была уверена
   в одной причине, а реальная причина оказалась другой.

### Сценарий 3: Доля правильных предсказаний

В пустой ячейке Excel:

```
=COUNTIF(R2:R36984;"✓")/COUNTA(R2:R36984)
```

(где `R` — буква колонки `cause_correct`). Получишь долю угаданных
причин. Аналогично для `binary_correct` — но там надо **исключить
«н/п»**:

```
=COUNTIF(Q2:Q36984;"✓")/(COUNTIF(Q2:Q36984;"✓")+COUNTIF(Q2:Q36984;"✗"))
```

### Сценарий 4: Матрица ошибок по причинам

Уже посчитана за тебя — лист `cause_confusion` в Excel-файле. Строки —
**реальная причина**, столбцы — **прогноз модели**. Диагональ — это
правильные предсказания, всё остальное — путаница. Пример чтения:

> в строке `погода`, в столбце `без задержки` стоит `2671` — значит
> 2671 рейс с реальной погодной задержкой модель ошибочно отнесла
> к категории «без задержки» (то есть пропустила задержку).

## Как модели были обучены (для справки в отчёте)

| Параметр | Голова A — задержка | Голова B — причина |
|---|---|---|
| Алгоритм | XGBoost | XGBoost |
| Набор признаков | `with_weather` (30 колонок: расписание + каскад + погода) | `with_weather` (то же самое) |
| Гиперпараметры | Optuna 30 trials (Run #6 в MLflow) | Optuna 30 trials (C5 в MLflow) + `compute_sample_weight('balanced')` |
| Метрика на val | F1 = 0.630, ROC-AUC = 0.839 | macro_f1 = 0.361, accuracy = 0.656 |
| Метрика на test | accuracy = **81.9 %** на оценимых рейсах | accuracy = **69.6 %** в целом |
| Random seed | 42 | 42 |

**Важно:** обучение шло **только** на данных до конца июня 2025
включительно. Тестовая выборка — рейсы с **1 июля 2025** и далее, то
есть **модель видит их впервые**. Это честная оценка «как модель
поведёт себя в проде».

## Важные оговорки

1. **Колонка `actual_*` — это «правда из будущего»**. В реальной
   системе на момент предсказания их не было бы. Они здесь только для
   того, чтобы мы могли посчитать, насколько модель права.

2. **Для отменённых рейсов (`cancellation_flag = 1` в исходных
   данных) колонка `predicted_delay` стоит `—`**, потому что у
   отменённого рейса вылета не было — задержка вылета не определена.
   Прогноз причины (`predicted_cause`) для них всё равно есть и его
   стоит сравнивать с `actual_cause = отменён`.

3. **Класс `cancelled` в причине** — это отменённые рейсы (отдельная
   причина в схеме). На них модель A не оценивается (прогноз = `—`),
   а модель B должна угадывать «отменён».

4. **`delay_probability` и `cause_confidence` — это вероятности**, а
   не «100 % правильности». Высокая уверенность ≠ гарантия попадания,
   просто модель чаще угадывает, когда уверенность выше.
